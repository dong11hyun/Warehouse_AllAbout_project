쿠버네티스

Bid 서비스가 대박이 나서 사용자가 100만 명이 되었다고 가정해 봅시다.

❌ 도커 컴포즈(지금 쓰는 방식)만 쓸 때의 문제점
과부하: 컴퓨터 한 대로는 web 컨테이너가 터져 나갑니다.

새벽 3시 장애: web 컨테이너가 갑자기 오류로 꺼졌습니다. 님은 자다가 일어나서 다시 켜야 합니다.

확장 불가: 컴퓨터를 10대로 늘렸습니다. 10대에 일일이 들어가서 docker run을 쳐줘야 합니다.

✅ 쿠버네티스를 쓸 때
님: "야, 쿠버네티스! web 컨테이너 50개 띄우고, 접속자 많아지면 100개까지 늘려. 그리고 하나라도 꺼지면 알아서 다시 살려놔."

쿠버네티스: "네, 알겠습니다." (알아서 여러 서버에 컨테이너를 분산 배치하고 감시함)


---

AWS는 **운동장(인프라)**을 제공합니다.

쿠버네티스는 그 운동장에서 뛰어노는 선수들(컨테이너)의 감독입니다.

선수가 너무 많아지면(컨테이너 오토스케일링), AWS가 운동장을 넓혀줍니다(서버 오토스케일링).

서로 경쟁하는 게 아니라, 서로 다른 층위(Layer)에서 돕는 관계입니다.
