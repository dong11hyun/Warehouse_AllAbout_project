
## 1. 아키텍처 패턴 (Architectural Patterns)
##### MVT(Model-View-Template)
- django 데이터(Model), 비즈니스 로직(View), 화면(Template)을 분리하여 관리

##### Service Layer
- `auctions/services.py`
    - 비즈니스 로직(입찰, 결제 등)을 View에서 분리하여 독자적인 계층으로 만들었습니다. 재사용성을 높이고 관심사를 분리합니다.
##### Broker Pattern
- (브로커 패턴)	Django Channels + Redis	실시간 통신을 위해 Consumer와 클라이언트 사이에서 메시지를 중개(Routing)하는 Redis(또는 In-Memory) 브로커를 사용합니다.

## 2. 디자인 패턴 (Design Patterns)
##### Composite
- `common/models.py` (Region)'지역'이 하위 '지역'을 포함하는 트리 구조(self 참조)를 형성합니다. 개별 객체(동)와 복합 객체(구/시)를 동일하게 다룹니다.
##### Observer / Pub-Sub
- (옵저버 / 발행-구독)`auctions/consumers.py`
group_send, group_add	특정 경매(auction_{id}) 그룹을 구독한 사용자들에게 상태 변화(입찰, 종료) 이벤트를 실시간으로 전파합니다.

##### Decorator
- @login_required
- @transaction.atomic	함수 코드를 수정하지 않고 '로그인 확인', '트랜잭션 묶기'라는 부가 기능을 동적으로 추가합니다.

##### Adaptor
- @database_sync_to_async	동기(Synchronous) 방식으로 작동하는 Django ORM 코드를 비동기(Asynchronous) 환경인 Channels에서 실행할 수 있도록 인터페이스를 변환합니다.

##### Factory Method
- User.objects.create_superuser
- get_channel_layer()	객체 생성 과정을 캡슐화한 메서드(Manager)를 통해 인스턴스를 생성합니다. 클라이언트는 구체적인 생성 로직을 몰라도 됩니다.

## 3. 엔터프라이즈 & 데이터베이스 패턴 (Enterprise & DB Patterns)
패턴 명	적용 코드 / 위치	설명
Active Record
(액티브 레코드)	models.py (모든 모델)	객체(Class)가 데이터(속성)와 행위(save, delete)를 모두 포함하며, DB 테이블의 레코드와 1:1로 매핑됩니다.
Unit of Work
(작업 단위)	with transaction.atomic():	"지갑 차감 + 입찰 기록 + 상태 변경" 등 일련의 작업을 하나의 원자적(Atomic) 단위로 묶어, 하나라도 실패하면 전체를 취소(Rollback)합니다.
Pessimistic Lock
(비관적 락)	select_for_update()	데이터 조회 시점에 DB Row에 잠금(Lock)을 걸어, 트랜잭션이 끝날 때까지 다른 사람이 수정하지 못하게 막는 강력한 동시성 제어 패턴입니다.

## 4. 웹 프레젠테이션 패턴 (Web Presentation Patterns)
패턴 명	적용 코드 / 위치	설명
Front Controller
(프론트 컨트롤러)	urls.py & Django Core	urls.py가 모든 요청을 가장 먼저 받아 적절한 View로 분배(Dispatch)하는 중앙 집중형 진입점 역할을 합니다.
PRG
(Post-Redirect-Get)	views.py (입찰 후)
return redirect(...)	POST(입찰) 요청 처리 후 Redirect를 보내 GET으로 페이지를 다시 부르게 합니다. (새로고침 시 중복 입찰 방지)
Flash Messages
(플래시 메시지)	messages.success(...)	사용자의 액션 결과("입찰 성공!")를 세션에 임시 저장했다가, 다음 화면(Redirect 된 화면)에서 한 번만 보여주고 삭제합니다.
